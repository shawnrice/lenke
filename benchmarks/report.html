pl-graph — does a Rust core earn its keep?

Short version: Rust wins decisively on throughput — graph build, traversal queries, and scans are one to two orders of magnitude faster. (The first run also surfaced a real bug: the TS executor overflowed the stack on big result sets; that's now fixed by streaming through the fp helpers, so the gap is speed, not capability.) The catch is a fixed ~4 ns FFI tax per call — so for tiny graphs or chatty per-element calls the boundary eats the win. The crossover: small graphs / interactive frontend use (stay TS) vs. bulk load + heavy query throughput (go Rust).

The TS executor's overflow was a bug — now fixed. The first run showed 1-hop@1M and 2-hop@100k failing outright: the executor materialized bindings between clauses and did push(...spread), overflowing the call stack. It now streams bindings lazily through the fp iterator helpers, so those queries complete and LIMIT n short-circuits (≈0.6 ms over 1M nodes). What's left is throughput, not capability: aggregation still buffers each group to fold it, so a large count(*) is correct but O(rows) — slower than Rust's columnar count, not a wall.

Verdict — when is the Rust core worth it?

Go Rust when: you're bulk-loading or querying large graphs (≳10k elements), running traversals or scans for throughput, or doing server-side materialization. Build is 7× faster and queries 10–359× faster — and serializing to disk/wire, Rust emits write-ready bytes while TS emits a string that still needs encoding.

Stay TS when: graphs are small (≲1k), or the workload is interactive/reactive frontend use where the graph lives in the browser, mutates constantly, and feeds React via the snapshot model. There the ~4 ns/call FFI tax dominates, and you'd lose the reactivity layer. (The TS engine now completes large queries too — just slower — so this is a performance choice, not a capability one.)

The boundary is the cost, not the compute. Rust compute is far faster everywhere; what claws it back is crossing FFI per call. So the architecture that wins is coarse-grained: hand Rust a whole NDJSON blob, let it build + query + aggregate, and pull back only small results — never chatty per-element calls. That's the Node/materialization persona; the frontend/reactive persona stays in TypeScript.
