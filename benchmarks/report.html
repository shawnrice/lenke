Short version: Rust wins decisively on throughput — graph build, traversal queries, and scans are one to two orders of magnitude faster, and the TS gql executor can't complete large traversals at all (it overflows the stack on big result sets). The catch is a fixed ~4 ns FFI tax per call and a marshalling cost on encode — so for tiny graphs or chatty per-element calls the boundary eats the win. The crossover is small graphs / interactive frontend use (stay TS) vs. bulk load + heavy query throughput (go Rust).

The TS engine has a hard ceiling. The gql executor accumulates bindings with out.push(...matched); on large traversal result sets that spread overflows the call stack, so 1-hop at ≥1M and 2-hop at ≥100k vertices fail outright in TS. The columnar Rust executor streams the DFS and has no such limit.

Verdict — when is the Rust core worth it?

Go Rust when: you're bulk-loading or querying large graphs (≳10k elements), running traversals or scans for throughput, or doing server-side materialization. Build is 7× faster, queries 10–315× faster, and — decisively — Rust completes large traversals the TS engine cannot.

Stay TS when: graphs are small (≲1k), or the workload is interactive/reactive frontend use where the graph lives in the browser, mutates constantly, and feeds React via the snapshot model. There the ~4 ns/call FFI tax and the encode copy-back dominate, and you'd lose the reactivity layer.

The boundary is the cost, not the compute. Rust compute is far faster everywhere; what claws it back is crossing FFI per call and copying bytes back on encode. So the architecture that wins is coarse-grained: hand Rust a whole NDJSON blob, let it build + query + aggregate, and pull back only small results — never chatty per-element calls. That's the Node/materialization persona; the frontend/reactive persona stays in TypeScript.
